Career-ops · 面试讲解流程图

系统串联逻辑 · 话术参考 · 工程亮点提炼  |  2026.04

1
输入层

JD 获取 · Playwright 爬取

用户输入招聘链接，Playwright webkit 无头浏览器自动渲染并提取 JD 文本，支持字节、腾讯、美团等 10+ 平台的动态渲染和反爬处理。也可直接粘贴 JD 文本跳过爬取。

2
缓存层

三级缓存 · 跳过重复 LLM 消耗

先查 URL 精确匹配（Level 1），再查 JD MD5 哈希（Level 2），最后用 sentence-transformers embedding 做 cosine 相似度检索（Level 3，阈值 0.92）。三级均未命中才进入 LLM。

命中则零 LLM 消耗
3
Stage 1

Archetype 分类 · 两阶段混合策略

关键词规则覆盖约 80% 场景（零 LLM 消耗）；置信度不足时回退 LLM 语义分类（~200 tokens）。分类结果决定 Stage 2 的维度权重配置和专属门控规则。

80% 场景 0 token
4
Stage 2

LLM 评估 · Structured Output

根据 Archetype 动态注入差异化权重，调用 LLM 对 7 个维度打分。Claude 后端用 tool_use 强制结构化输出，直接解析为 typed dict；Ollama 用 format=json。parse 失败率从 ~8% 降至 ~0。

parse 失败率 ~8% → ~0
5
门控层

Gate-Pass Logic · 业务规则约束

LLM 输出后叠加业务约束层：岗位匹配度 < 35 强制压至 C，经验门槛过高压至 D；大模型岗匹配度 < 50 压至 B。多条门控同时触发取最严格上限，记录 original_grade 和触发原因。

防止单维高分掩盖短板
6
输出层

PDF 报告 · 看板 · STAR 故事

评估结果写入 jobs.json，生成带维度进度条的中文 PDF，刷新 HTML 可视化看板。加 --star 参数时额外调用 LLM 生成面试 STAR 素材并追加到 story_bank.md。

7
质量保障

评估集 · A/B 测试 · 回归

10 个 Golden Label case 覆盖全等级及对抗 case，eval_runner.py 自动回归，等级准确率硬指标 10/10。A/B 测试框架量化 Token 优化代价：评分偏差 ≤3 分，Token 节省 27%。

等级准确率 10/10 Token 节省 27%

面试话术参考

请介绍一下你的 Career-ops 项目
这是一个我从零独立设计开发的大模型驱动的求职自动化系统。核心功能是：给一个招聘链接，系统自动爬取 JD、评估匹配度、生成 PDF 报告，整个流程全自动。但我想重点讲的不是"功能是什么"，而是我是怎么做工程决策的。

整个系统分三层来看：输入层用 Playwright 做多平台爬取；评估层是我设计的两阶段 LLM Pipeline；质量保障层包括三级缓存、结构化输出、门控机制和评估集。
两阶段 Pipeline 具体是怎么串联起来的？
JD 进来之后，先过三级缓存——URL 精确匹配、MD5 哈希、最后是 embedding 语义相似度检索。命中任何一级就直接返回，不进 LLM，这样大概 30% 的重复 JD 零消耗。

缓存未命中才进 Pipeline。1Stage 1 做 Archetype 分类，先用关键词规则，能覆盖约 80% 的明确 JD，完全不耗 token；剩下 20% 模糊的回退 LLM 做语义分类，大概 200 tokens。

2分类结果驱动 Stage 2——不同岗位类型注入不同的维度权重。比如大模型岗的岗位匹配度权重是 30%，普通岗位是 25%，让评分更贴近这个岗位真正重要的维度。

3LLM 打完分之后，过门控层——这是我加在 LLM 输出之后的业务规则约束，相当于对 LLM 结果做后处理。核心逻辑是：某个关键维度低于阈值，不管其他维度多高，整体等级都要被压下来，防止一个维度高分掩盖真实短板。
你怎么保证 LLM 输出的稳定性？
两个层面。

输出格式的稳定性：我没有用正则去解析 LLM 的自由文本，而是用 Claude 的 tool_use + tool_choice 强制模型按 JSON Schema 输出结构化结果，直接拿到 typed dict，parse 失败率从之前 ~8% 降到接近零。Ollama 那边用 format=json 保持接口一致。

评分质量的稳定性：我做了 A/B 测试，同一个 JD 跑两个 Variant——全文 CV 对比摘要 CV，各跑 N 轮取均值消除 LLM 随机性。实测摘要 CV 压缩 69% 之后，评分偏差只有 ≤3 分，说明优化是有效的。另外我还维护了一套 Golden Label 评估集，10 个 case 覆盖全部等级和对抗 case，每次改 Prompt 模板都要跑一遍回归，等级准确率必须 10/10 才能合并。
门控机制有什么具体设计？
我把门控分成两层：全局门控和 Archetype 专属门控。

全局门控对所有岗位生效，比如岗位匹配度 < 35 强制压到 C 级，因为方向不符申请价值极低。经验要求匹配度 < 20 压到 D 级，说明门槛对实习生太高了。

Archetype 专属门控更细，比如大模型岗位，匹配度 < 50 就压到 B，因为这个方向专业壁垒高，方向不符的话即使其他维度不错也没有价值。

多条门控同时触发的时候，取最严格的那个上限，同时把原始等级和触发原因都记下来，方便用户看到"为什么被压"。
